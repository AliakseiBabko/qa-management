# Document Contract

Primary final output is a Google Sheet per project in `20_M2_Project_Management\<Project>`,
with local CSV fallback, plus one workspace-wide rollup Sheet. Preserve the
CSV template columns as the Sheet schema.

## Purpose

Use this reference for the M2 timeline/action-items document family: tracking
dated events (meetings, status-report commitments, deadlines, follow-ups/
clarifications) so nothing gets forgotten across multiple projects, and
seeing what's due today/tomorrow/this week in one place.

## Template

`<repo-root>\Templates\action_items.csv`

## Expected Output

- One `action_items` Sheet per project (living list, not a dated snapshot).
- One workspace-wide `_timeline` Sheet directly under `20_M2_Project_Management`,
  next to `_project_registry`/`_people_registry` — a mechanical rollup of every
  project's open (`Статус = Открыто`) `action_items` rows, sorted by
  `Дата события`.

Suggested target folder for the per-project Sheet:

`G:\My Drive\QA_Management\20_M2_Project_Management\<Project>`

Local CSV fallback filename: `action_items.csv`.

## Versioning

- `action_items` is a living canonical file: edit rows in place (date slips,
  status changes) rather than appending duplicates or creating dated
  snapshots. This is unlike `evidence_log`, which is append-only.
- `_timeline` is fully generated by `refresh_timeline_registry.py` from every
  project's `action_items`. Never edit `_timeline` directly — an edit there
  is silently overwritten on the next refresh.
- `scaffold_project_dashboard.py` creates an empty `action_items` Sheet
  (header row only) for a project if one doesn't exist yet; it never
  overwrites an existing one.

## Schema

Use exactly the columns in `Templates\action_items.csv`:

1. `Проект`
2. `Дата события` — ISO `YYYY-MM-DD`. Never blank; if the real date is
   unknown, use the nearest concrete placeholder and record the uncertainty
   in `Комментарии`.
3. `Тип` — free text but keep to a small consistent set: `Встреча`
   (1:1/status/client sync), `Отчёт/статус в чат`, `Дедлайн`, `Follow-up`
   (a clarification owed to or from someone), `Прочее`.
4. `Что нужно сделать` — concrete, one action, not a vague intention.
5. `Статус` — `Открыто`, `Выполнено`, or `Отменено`.
6. `Owner` — who acts on it: `M2`, a named QA, or the client side. Never blank.
7. `Источник` — same discipline as `evidence_log`: name the chat/transcript/
   meeting that produced the item.
8. `Комментарии` — optional context, including any date-uncertainty note.

`_timeline` uses the same columns as `action_items` (all projects
concatenated, `Статус = Открыто` rows only, sorted by `Дата события`).

## Inputs

- Meeting transcripts and strategy/status chats — any concrete next meeting,
  deadline, or promised follow-up mentioned there is a candidate row.
- `m2_input` rounds — a question still owed an answer, or an answer that
  implies a next step with a date.
- Direct conversational instructions from M2 ("remind me to ask X on the
  next call", "report is due Friday").
- `scan_open_questions.py` — mechanically surfaces candidates from
  `m2_input` (pending rounds), `project_risk` (`План действий`), and
  `project_metrics` (`Неизвестно` rows) across all projects at once. Always
  review/rewrite its placeholder `Тип`/`Owner`/`Дата события` before
  logging (see `SKILL.md`, "Deriving action items from project state") —
  it surfaces gaps, it does not decide the right next action.

## Evidence Rules

- Log only concrete, datable items — not general risk or health commentary
  (that belongs in `project_risk`).
- Keep one row per action; don't bundle multiple unrelated to-dos into one
  `Что нужно сделать` cell.
- Close (`Выполнено`/`Отменено`) rather than delete resolved items — the
  closed history is still useful context for the next 1:1 or status report,
  even though it's no longer surfaced in `_timeline`.

## Rule

Keep this skill scoped to the timeline/action-items document family only.
Do not use it to record risk judgment (`project_risk`) or the append-only
source trail (`evidence_log`).
