# M2 Project Development Plan

Use this skill for one output family only:

- project-level development-plan document

## Required Start

1. Read `references/document-contract.md`.
2. Identify the target project, period, review cycle, and next review date if present.
3. Read project development plans first, then project metrics, risk summaries, and workbook status rows.

## Workflow

1. Produce one row per project development focus or initiative.
2. Include the current state, why the initiative matters, action, owner, due date, and success criterion.
3. Keep the plan management-oriented: process, staffing, QA visibility, automation/regression, documentation, communication, and delivery predictability.
4. Use evidence from the source corpus; keep uncertainty explicit.

## Guardrails

- Do not mix this output with metrics output.
- Do not mix this output with project-risk output.
- Do not turn an individual employee development issue into a project initiative unless it materially affects project delivery or QA process.
