# M2 Project Development Plan

Use this skill for one output family only:

- project-level development-plan document

## Required Start

1. Read `references/document-contract.md`.
2. Read `../qa-management-roles/references/m2-role-rules.md`.
3. Identify the target project, period, review cycle, and next review date if present.
4. Read project development plans first, then project metrics, risk summaries, and workbook status rows.

## Workflow

1. Produce one row per project development focus or initiative.
2. Start from business/project context: how the project creates value, what success means, and what the client/business needs.
3. Include the current state, why the initiative matters, action, owner, due date, and success criterion.
4. Cover project movement, business/client value, QA/process value, staffing/continuity, communication, and role/value growth where evidence supports it.
5. Use evidence from the source corpus; keep uncertainty explicit.

## Guardrails

- Do not mix this output with metrics output.
- Do not mix this output with project-risk output.
- Do not turn an individual employee development issue into a project initiative unless it materially affects project delivery or QA process.
- Do not treat a QA task list, automation plan, or completed-work report as a project development plan unless it is mapped to project/business goals and success criteria.
