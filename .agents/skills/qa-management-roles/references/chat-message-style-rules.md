# Chat Message Style Rules

For any short chat message drafted for the user to send to a colleague or
stakeholder — a 1:1 invite, a feedback request, a status ping, a
first-contact message — not for the longer analytical documents this
workspace generates (development plans, risk reports, status reports),
which have their own tone.

See `google-workspace-rules.md`'s Language Rules for the em-dash rule,
which applies here too.

## Rules

- **Don't add a clarifying question the user didn't ask for**, especially
  about org structure or roles. The user drafting the message already
  knows who the recipient is and what they do; don't manufacture a
  clarifying question on their behalf just because a role looks ambiguous
  from the documents alone.
- **No meta-commentary explaining the message's own framing or intent**
  inside the message itself (e.g. "this isn't a status report," "not in
  the sense of an obligation"). Let the plain, direct tone carry that
  instead of spelling it out.
- **Don't default to an explicit "meeting or short message, your choice"
  style closing.** Leave the message open-ended and let the recipient pick
  their own format, unless the user specifically asks for that kind of
  closing on a given message.
- **When the user provides their own edited variant of a drafted message,
  use their wording close to verbatim** in future messages of the same
  kind, rather than re-paraphrasing back toward the original draft.
- General shape that works: short direct opener (who you are + why you're
  writing) → 2-4 topics, bulleted if there's more than one → one sentence
  on why the input matters/what it's for → no closing meta-ask unless
  specifically requested.

## Rationale

These came from repeated corrections across several rounds of drafting
real chat messages: the first draft kept adding hedges, clarifying
questions, and closing disclaimers the user didn't want, made the message
read as over-explained rather than natural, and got edited back down each
time in the same direction.
